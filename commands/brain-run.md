Сценарий: Roo Code или пользователь вносил изменения напрямую.

### C1 — Собрать изменения

Параллельно:
```bash
git status --short
git diff --stat HEAD
```
`mcp__ide__getDiagnostics` — TS/ESLint ошибки.

### C2 — Найти активный план

```bash
grep -rl "🔄 В работе" plans/
```

### C3 — Авто-матч diff → шаги

Для каждого изменённого файла найти шаги с этим файлом в поле **Файлы**:
- изменён + нет TS-ошибок → `[x] Выполнено`
- изменён + TS-ошибки → `[~] Ошибки: <список>`
- не тронут → `[ ]`
- Изменения вне плана → **внеплановые**


## Протокол выполнения

### Шаг 1 — Найти план

**Проверить `$ARGUMENTS`:**

- **`$ARGUMENTS` непустой** (путь к файлу, `шаг N M`, текстовое описание) → **grep не запускать**.
  - Содержит `plans/...md` → читать этот файл напрямую.
  - Содержит `шаг N M` или текст → план уже известен из контекста беседы; если нет — спросить: `Какой план? Укажи файл: plans/<name>.md`
- **`$ARGUMENTS` пустой** → искать активный план:
  `Grep("🔄 В работе", "plans/*.md", files_with_matches)` — **не читать** `✅ Выполнено`.
  Если несколько — первый по дате в имени.
  Если нет — ответить: `Нет активного плана. Создай: /brain-plan <задача>`

Прочитать план полностью. Извлечь:
- Шаги `[ ] Не начат`
- Поля **Файлы**, **Результат**, **Проверка**, **Агент**
- `## Параллельные шаги`
- Зависимости `⬆ После шага N`

Если существует `C:\Users\user\ВОблако\brain\projects\vitejs-vite-6hpxonsb\known-issues.md` — прочитать, выделить записи с файлами/тегами из плана:
```
⚠️ Известные проблемы:
  - #supabase: NULL в поле X требует явного приведения
```

### Шаг 2 — Граф выполнения и ПОДТВЕРЖДЕНИЕ

Вывести пользователю:

⚠️ **СТОП.** Не начинать выполнение шагов.
Ждать подтверждения пользователя: 'да', 'go', 'начинай', 'запускай'.
Только после явного подтверждения → переходить к Шагу 2б.

### Шаг 2б — UI-гард (если шаг помечен ⚠️ UI)

Для каждого шага с полем `**UI:** ⚠️` — **до** спавна субагента:

1. Прочитать `src/styles/uiTokens.ts` — выписать значения затронутых групп токенов
2. Проверить FROZEN: `git diff HEAD -- <файлы> | grep "positionCardAwayFromConnections\|MobileConnectionGraph\|card\.mobile\|parchment"`
3. Показать пользователю:
```
⚠️ UI-CRITICAL: <название шага>
Токены: UI_TOKENS.<группа> → <что изменится визуально>
FROZEN в этом шаге: нет / 🔒 <что под риском>
Визуально: <описание ожидаемого изменения на экране>
```
4. ⚠️ **СТОП** — ждать явного подтверждения: `да` / `go` / `proceed`
   Только после — запускать субагента.

### Шаг 2в — Hermes ctx-lookup перед спавном

Для каждого шага, который будет запущен — извлечь ключевую тему из названия/описания шага и вызвать `/wiki <тема>`. Контент инжектировать в промпт соответствующего субагента вместо перечисления файлов.

Если несколько шагов параллельные с одной темой — один `/wiki <тема>` на группу.
Если тема шага неочевидна → `/wiki entry` (fallback на map.md).

### Шаг 4 — Верификация

```bash
git diff --stat HEAD
git status --short
```
`mcp__ide__getDiagnostics` — TS/ESLint ошибки в изменённых файлах.

- Агент ВЫПОЛНЕНО + нет TS-ошибок → `[x]`
- Агент ВЫПОЛНЕНО + есть TS-ошибки → `[~]`
- Агент ЗАБЛОКИРОВАНО → `[!]`

**Если шаг был ⚠️ UI — дополнительная проверка:**
```bash
# Хардкод в компонентах (не uiTokens.ts) → пометить [~]
grep -n "rgba(\|blur(\|#[0-9a-fA-F]\{3,6\}" <изменённые UI-файлы>
# Нормально: только в src/styles/uiTokens.ts
# Нарушение: rgba/hex прямо в .tsx-компонентах

# FROZEN не задет?
git diff HEAD -- <файлы> | grep -i "positionCardAwayFromConnections\|MobileConnectionGraph\|card\.mobile\|parchment"
# Ожидается: пусто
```
Если grep нашёл хардкод в компоненте → пометить `[~] UI-токен нарушен`, не ставить `[x]` до исправления.

## Правила качества

- `[x]` только если нет TS-ошибок в изменённых файлах (getDiagnostics)
- При конфликте файлов между параллельными агентами → `isolation: "worktree"`
- Никогда не ставить `[x]` без проверки getDiagnostics
- Планы `✅ Выполнено` — **архив, не читать**
