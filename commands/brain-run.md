# /brain-run — Работа по плану

<!--
  НАСТРОЙКА ПОД СВОЙ ПРОЕКТ
  ══════════════════════════
  Перед использованием замени все блоки, помеченные [НАСТРОИТЬ]:
  1. Стек и путь проекта в промпте агента
  2. Критические инварианты (что нельзя трогать)

## Пути

```
plans/       ← файлы планов
```

**WIKI_DIR:** `<your_brain_wiki_dir>`  ← [НАСТРОИТЬ]

## Режимы

| Команда | Действие |
|---------|----------|
| `/brain-run` | Все незакрытые шаги активного плана |
| `/brain-run шаг 2 3` | Только шаги 2 и 3 |
| `/brain-run plans/2026-04-30_foo.md` | Конкретный план |
| `/brain-run check` | Засчитать прямые изменения (без агентов) |

---

## Режим CHECK (`/brain-run check`)

Сценарий: изменения внесены вручную или другим инструментом.

### C1 — Собрать изменения

```bash
git status --short
git diff --stat HEAD
```

### C2 — Найти активный план

```
Grep("🔄 В работе", "plans/*.md", files_with_matches)
```

### C3 — Авто-матч diff → шаги

Для каждого изменённого файла найти шаги с этим файлом в поле **Файлы**:
- изменён + нет ошибок → `[x] Выполнено`
- изменён + есть ошибки → `[~] Ошибки: <список>`
- не тронут → `[ ]`
- изменения вне плана → **внеплановые**

### C4 — Обновить план, TaskUpdate, вывод

```
🔍 Изменено файлов: N
✅ Засчитано: [x] Шаг N
⚠️ С ошибками: [~] Шаг M
📦 Внеплановые: path/to/file
Прогресс: ████████░░ 3/4
```

## Протокол выполнения

### Шаг 1 — Найти план

Проверить `$ARGUMENTS`:
- Содержит `plans/...md` → читать этот файл напрямую
- Содержит `шаг N M` → план известен из контекста беседы
- Пустой → `Grep("🔄 В работе", "plans/*.md", files_with_matches)`
  - Если несколько → первый по дате в имени
  - Если нет → `Нет активного плана. Создай: /brain-plan <задача>`

Прочитать план. Извлечь:
- Шаги `[ ] Не начат`
- Поля **Файлы**, **Результат**, **Проверка**, **Агент**
- `## Параллельные шаги` и зависимости `⬆ После шага N`

Если есть `<your_brain_wiki_dir>/known-issues.md` — прочитать, выделить записи релевантные файлам/тегам плана.

### Шаг 2 — Граф выполнения и ПОДТВЕРЖДЕНИЕ

```
🗺 plans/<filename>.md — N незакрытых шагов

Параллельно: Шаг 1, Шаг 2
После них → Шаг 3
```

⚠️ **СТОП.** Ждать подтверждения: 'да', 'go', 'начинай', 'запускай'.

### Шаг 2б — Wiki ctx-lookup перед спавном

Для каждого шага — извлечь тему из названия/описания и вызвать `/wiki <тема>`.
Контент инжектировать в промпт субагента вместо перечисления файлов.

Если несколько параллельных шагов с одной темой → один `/wiki <тема>` на группу.
Если тема неочевидна → `/wiki entry` (fallback на главный файл wiki).

### Шаг 3 — Запустить агентов

Для каждого шага — `Agent(subagent_type: "general-purpose")`.

Параллельные шаги → несколько `Agent` в одном сообщении.
Последовательные → строго после завершения предыдущей группы.

Промпт агента (self-contained — агент не видит контекст беседы):

```
Ты работаешь в проекте: [НАСТРОИТЬ — стек, например: Node.js + TypeScript + PostgreSQL]
Путь к проекту: <your_project_path>

КРИТИЧЕСКИЕ ИНВАРИАНТЫ:
[НАСТРОИТЬ — что нельзя нарушать. Примеры:
- Не коммитить .env и секреты
- Не менять <критичная часть> без явного указания
- Миграции только вперёд]

Контекст wiki (из /wiki <тема>):
[содержимое файлов, полученных на Шаге 2б]

Известные блокеры (из known-issues.md):
[релевантные записи, если есть]

ЗАДАЧА (Шаг N — <название>):
<полный текст шага>

Файлы: <из поля Файлы>
Результат: <из поля Результат>
Критерий: <из поля Проверка>

Верни отчёт:

## ОТЧЁТ: Шаг N

**Статус:** ВЫПОЛНЕНО | ЧАСТИЧНО | ЗАБЛОКИРОВАНО

**Изменённые файлы:**
- path/to/file (строки X-Y: что изменено)

**Что сделано:** <краткое описание>

**Критерий выполнен:**
- [x] <критерий>
- [ ] <не выполнен — причина>

**Блокеры:** <если есть>

**Для solutions.md:**
- **Проблема:** ... **Решение:** ... **Файлы:** ... **Теги:** #tag

**Для .raw_nuances.md:**
- **Нюанс:** ... **Контекст:** ... **Теги:** #tag
```

### Шаг 4 — Верификация

```bash
git diff --stat HEAD
git status --short
```

Запустить проверку качества кода для твоего стека:
```bash
# [НАСТРОИТЬ — примеры:]
# TypeScript:  npx tsc --noEmit
# Python:      python -m pytest  /  ruff check .
# Rust:        cargo check
# Go:          go build ./...
# Ruby:        bundle exec rubocop
```

- Агент ВЫПОЛНЕНО + нет ошибок → `[x]`
- Агент ВЫПОЛНЕНО + есть ошибки → `[~]`
- Агент ЗАБЛОКИРОВАНО → `[!]`

### Шаг 4б — Сохранить knowledge

Из отчётов агентов записать в wiki:

**solutions.md** — нетривиальные проблемы с решениями:
```markdown
## <YYYY-MM-DD> | <plans/filename.md> | Шаг N «название»
**Проблема:** ...
**Решение:** ...
**Файлы:** path/to/file:строка
**Теги:** #тег
```

**.raw_nuances.md** — архитектурные инварианты и неочевидные паттерны:
```markdown
## <YYYY-MM-DD> | <plans/filename.md> | Шаг N «название»
**Нюанс:** ...
**Контекст:** ...
**Файлы:** path/to/file
**Теги:** #тег
```

После записи — wiki-watch авто-пушит изменения (если запущен).
Если не запущен — вручную:
- Windows: `python wiki-push.py` (из папки brain/)
- macOS/Linux: `python3 wiki-push.py` (из папки brain/)

### Шаг 5 — Обновить план-файл

```markdown
- **Статус:** [x] Выполнено <дата>
```

В `## Лог выполнения`:
```markdown
**<дата> /brain-run** — Шаг N «<название>»
- Итог: ВЫПОЛНЕНО / ЗАБЛОКИРОВАНО
- Файлы: <список>
- Заметки: <из отчёта>
```

Если все `[x]` → `🔄 В работе` → `✅ Выполнено`.

### Шаг 6 — TaskUpdate

`TaskList` → найти `[план]`-таски выполненных шагов → `TaskUpdate` completed.

### Шаг 7 — Финальный вывод

```
🏁 /brain-run завершён
📋 plans/<filename>.md

✅ Выполнено:
  [x] Шаг 1 — <название>

[~] С ошибками:
  [~] Шаг 2 — <ошибка>

[!] Заблокировано:
  [!] Шаг 3 — <блокер>

⏳ Не запускалось:
  [ ] Шаг 4 (зависело от шага 3)

Прогресс: ████████░░ 4/5
Изменено файлов: N

→ Если изменено файлов src/: запусти /wiki
→ Если шаги [!]: /brain-plan <блокер>
```

---

## Правила качества

- `[x]` только после успешной проверки (tsc / pytest / cargo check — см. Шаг 4)
- При конфликте файлов между параллельными агентами → `isolation: "worktree"`
- Планы `✅ Выполнено` — архив, не читать
