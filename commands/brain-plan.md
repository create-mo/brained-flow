# /brain-plan — Детальное планирование

<!--
  НАСТРОЙКА ПОД СВОЙ ПРОЕКТ
  ══════════════════════════
  Перед использованием замени все блоки, помеченные [НАСТРОИТЬ]:
  1. Стек проекта (раздел "Контекст для Plan-агента")
  2. Критические инварианты (что нельзя трогать в твоём проекте)
  3. Ключевые слова для тем → топик-маппинги под твой стек
  4. Путь к wiki (WIKI_DIR)

## Пути

```
plans/           ← файлы планов (относительно корня проекта)
```

**WIKI_DIR:** `<your_brain_wiki_dir>`  ← [НАСТРОИТЬ: абсолютный путь к твоей brain/ папке]

## Режимы

| Команда | Действие |
|---------|----------|
| `/brain-plan <задача>` | Создать новый план |
| `/brain-plan о` | Показать статус всех активных планов |

---

## Режим `/brain-plan о` — Проверка статуса

1. `Grep("🔄 В работе", "plans/*.md", output_mode: "files_with_matches")` — **не читать** файлы с `✅ Выполнено`
2. Для каждого найденного файла прочитать и посчитать `[x]` и `[ ]`

```
📊 Активные планы:

1. plans/2026-04-30_название.md
   Прогресс: ██████░░░░ 3/5 шагов
   [ ] Шаг 4 — ...
   [ ] Шаг 5 — ...

Итого: N активных | M шагов открыто
```

---

## Режим `/brain-plan <задача>` — Создание плана

### Шаг 1 — Загрузка контекста из wiki

**Извлечь тему задачи** из описания — 1-2 ключевых слова:

```
[НАСТРОИТЬ — примеры под свой стек:]
«починить авторизацию» → auth
«добавить уведомления» → notifications
«проблема с базой данных» → database
«ошибка в рендеринге» → render
```

**Вызвать `/wiki <тема>`:**
```
/wiki <извлеченная тема>
```

Если тема не ясна:
```
/wiki entry   # fallback на главный файл wiki
```

`/wiki <тема>` вернёт 2-3 релевантных файла — экономия ~60% контекста против чтения всей wiki.

**Проверить активные планы:**
```
Grep("🔄 В работе", "plans/*.md", files_with_matches)
```
Файлы с `✅ Выполнено` — **не читать**.

**Проверить known-issues.md** если вернулся из `/wiki` — искать релевантные блокеры для темы.

**Проверить актуальность wiki:**
```
/wiki --status
```
Если устарела → предупредить: "Wiki устарела (N коммитов). Рекомендую `/wiki` перед планированием."

### Шаг 2 — Запуск Plan-агента

Запустить `Agent(subagent_type: "Plan")`:

```
Задача: <задача пользователя>

Контекст проекта (из /wiki <тема>):
[содержимое файлов, возвращённых /wiki]

Стек: [НАСТРОИТЬ — например: Node.js + TypeScript + PostgreSQL + React]

Критические инварианты проекта:
[НАСТРОИТЬ — что нельзя трогать, ключевые ограничения. Примеры:
- Не коммитить .env / секреты
- Не менять <критичная функция> без согласования
- Миграции только вперёд, не откатывать]

Известные решения (из known-issues.md, если был возвращён):
[релевантные записи]

Нужен план:
1. Разбивка на конкретные шаги (каждый — одна задача за сессию)
2. Для каждого шага: затронутые файлы, ожидаемый результат
3. Порядок: что параллельно, что строго последовательно
4. Критерий проверки каждого шага
5. Общий критерий "задача выполнена"
6. Риски и митигации
7. Если решение уже есть в known-issues.md → указать в плане
```

### Шаг 3 — Создание файла плана

Создать файл: `plans/<YYYY-MM-DD>_<slug>.md`
- Дата из системного контекста `# currentDate`
- Slug: 3-5 слов из задачи, латиница, дефисы, lowercase

```markdown
# <Название задачи>
> Создан: <дата> | Статус: 🔄 В работе

## Цель
<одна фраза>

## Критерий готовности
<как проверить завершённость>

## Шаги

### Шаг 1 — <название>
- **Агент:** general-purpose
- **Файлы:** <список файлов>
- **Результат:** <что появится/изменится>
- **Проверка:** <как верифицировать>
- **Статус:** [ ] Не начат

### Шаг 2 — <название>
- ⬆ После шага 1
...

## Параллельные шаги
<какие шаги можно запустить одновременно>

## Риски
- <риск>: <митигация>

## Лог выполнения
<!-- сюда пишем заметки по ходу работы -->
```

### Шаг 4 — Создание тасков

Для каждого шага вызвать `TaskCreate`:
- `name`: `"[план] Шаг N — <название>"`
- `description`: описание шага + `\n\nФайл плана: plans/<filename>.md`

### Шаг 5 — Вывод и СТОП

```
📋 План: plans/<filename>.md
📌 Тасков создано: N

Шаги:
1. [ ] <шаг 1>
2. [ ] <шаг 2>
...
```

⚠️ **СТОП.** `/brain-plan` только СОЗДАЁТ план. Не начинать выполнение.
Пользователь сам запустит `/brain-run` когда будет готов.

## Ключевые слова для извлечения темы

[НАСТРОИТЬ — замени на слова своего проекта]

```
auth, login, token, session    → auth
database, migration, schema    → database
api, endpoint, request         → api
render, ui, component, style   → frontend
deploy, ci, build, docker      → devops
test, spec, coverage           → testing
```

---

## Правила

- Каждый шаг выполним за одну сессию
- Если > 8 шагов — разбить на Фазы
- Зависимости: "⬆ После шага N"
- Файлы с `✅ Выполнено` — архив, не читать
- Экономия контекста: `/wiki <тема>` вместо чтения всей wiki (3 файла вместо всех → ~60% экономии)
