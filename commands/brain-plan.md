# /brain-plan — Детальное планирование

## Путь к планам

**ВСЕГДА** сохранять файлы в:
```
<your_project_path>\plans\
```

## Режимы

| Команда | Действие |
|---------|----------|
| `/brain-plan <задача>` | Создать новый план |
| `/brain-plan о` | Показать статус всех активных планов |

---

## Режим `/brain-plan о` — Проверка статуса

1. `Grep("🔄 В работе", "plans/*.md", output_mode: "files_with_matches")` — **не читать** файлы с `✅ Выполнено`
2. Для каждого найденного файла прочитать и посчитать `[x]` и `[ ]`

```
📊 Активные планы:

1. plans/2026-04-30_foo.md
   Прогресс: ██████░░░░ 3/5 шагов
   [ ] Шаг 4 — ...
   [ ] Шаг 5 — ...

Итого: N активных | M шагов открыто
```

---

## Режим `/brain-plan <задача>` — Создание плана

### Шаг 1 — Анализ задачи с Hermes-памятью

**Извлечь тему задачи:**
- Из описания задачи извлечь ключевую тему:
  - «починить VexFlow stems» → `vexflow`
  - «добавить audio source» → `audio_source`
  - «вернуть layout для medusa» → `medusa`
  - «проблема с YouTube» → `youtube`
  - «ошибка в useAudioPlayer» → `audio`

**Вызвать `/wiki <тема>`:**
```bash
/wiki <извлеченная тема>
```

**Если тема не ясна или задача общая:**
```bash
/wiki entry  # fallback на map.md
```

**Получить контент:**
- `/wiki <тема>` вернет 2-3 релевантных файла (ctx-режим)
- Вместо чтения 8+ файлов напрямую → экономия ~60% контекста
- Агент получит только нужный контент для планирования

**Активные планы:**
```bash
Grep("🔄 В работе", "plans/*.md", files_with_matches)
```
**НЕ читать** файлы с `✅ Выполнено`

**Проверка wiki:**
Если wiki устарела (есть изменения в `src/` новее wiki-файлов) — проверить и предупредить:
```bash
/wiki --status
```
> "Wiki устарела (N коммитов). Рекомендую `/wiki` перед планированием."

**Проверка known-issues:**
Если `/wiki <тема>` вернул `known-issues.md` → искать релевантные блокеры для темы задачи.

**UI-классификация шагов:**

Проверить: задача затрагивает UI-файлы?

UI-файлы проекта: `src/components/ComposerOverlay.tsx`, `src/components/ScoreCanvas.tsx`, `src/styles/uiTokens.ts`, любой компонент с `import.*UI_TOKENS`.

Если да → для каждого шага, касающегося этих файлов:
- Пометить `⚠️ UI` в поле **UI** шага
- Перечислить группы затрагиваемых токенов (`glass.*`, `card.hover.*`, `text.*`, ...)
- Проверить не затрагивает ли шаг FROZEN-список

**FROZEN — никогда не трогать:**
- `positionCardAwayFromConnections()` — ScoreCanvas.tsx / ComposerOverlay.tsx
- `MobileConnectionGraph` (Medusa)
- Mobile `SelectedCard` (~строки 370-490 ComposerOverlay.tsx)
- `UI_TOKENS.card.mobile`
- `UI_TOKENS.parchment` (Canvas API, не CSS)

Если шаг **пересекается с FROZEN** → добавить явное предупреждение в план и пометить `🔒 FROZEN-риск`.

**Контекст для Plan-агента:**
```
Контекст проекта (через /wiki <тема>):
- [файлы, возвращенные /wiki]
- React 19 + TypeScript + Vite 7 + PixiJS + Supabase
- КРИТИЧНО: не менять computeConcentricArcLayout (src/utils/layout.ts), не коммитить .env
```

### Шаг 2 — Запуск Plan-агента с Hermes-контекстом

Запустить `Agent(subagent_type: "Plan")`:

```
Задача: <задача пользователя>

Контекст проекта (из /wiki <тема>):
- [содержимое файлов, возвращенных /wiki]
- React 19 + TypeScript + Vite 7 + PixiJS + Supabase
- КРИТИЧНО: не менять computeConcentricArcLayout (src/utils/layout.ts), не коммитить .env

Известные решения (из known-issues.md, если был возвращен):
- [извлечь релевантные решения из known-issues.md]

Нужен план:
1. Разбивка на конкретные шаги (каждый — один агент за сессию)
2. Для каждого шага: затронутые файлы, ожидаемый результат
3. Порядок: что параллельно, что строго последовательно
4. Критерий проверки каждого шага
5. Общий критерий "задача выполнена"
6. Риски и митигации
7. Если решение уже есть в known-issues.md → указать это в плане
```

**Важно:** не передавать весь wiki в Plan-агента. `/wiki <тема>` уже отфильтровал контент.

**Экономия:** вместо 16k токенов wiki → 6k токенов тематического контекста.

### Шаг 3 — Создание файла плана

Создать файл: `plans/<YYYY-MM-DD>_<slug>.md`
- Дата из системного контекста `# currentDate`
- Slug: 3-5 слов из задачи, латиница, дефисы, lowercase

```markdown
# <Название задачи>
> Создан: <дата> | Статус: 🔄 В работе

## Цель
<одна фраза>

## Критерий готовности
<как проверить завершённость>

## Шаги

### Шаг 1 — <название>
- **Агент:** <кто выполняет>
- **Файлы:** <список>
- **Результат:** <что появится/изменится>
- **Проверка:** <как верифицировать>
- **UI:** нет / ⚠️ `UI_TOKENS.<группа>` — только через токены, не хардкод
- **FROZEN:** нет / 🔒 `<что не трогать>`
- **Статус:** [ ] Не начат

### Шаг 2 — <название>
...

## Параллельные шаги
<какие шаги можно запустить одновременно>

## Риски
- <риск>: <митигация>

## Лог выполнения
<!-- сюда пишем заметки по ходу работы -->
```

### Шаг 4 — Создание тасков

Для каждого шага вызвать `TaskCreate`:
- `name`: `"[план] Шаг N — <название>"`
- `description`: описание шага + `\n\nФайл плана: plans/<filename>.md`

### Шаг 5 — Вывод и СТОП

```
📋 План: plans/<filename>.md
📌 Тасков создано: N

Шаги:
1. [ ] <шаг 1>
2. [ ] <шаг 2>
...
```

⚠️ **СТОП.** `/brain-plan` только СОЗДАЁТ план. Не начинать выполнение.
Пользователь сам запустит `/brain-run` когда будет готов.

## Правила

- Каждый шаг — выполнимо за одну сессии
- Если > 8 шагов — разбить на Фазы
- Зависимости: "⬆ После шага N"
- Файлы с `✅ Выполнено` — архив, не читать

## Hermes-память (Graph-augmented Retrieval)

**Критично:** `/brain-plan` использует `/wiki <тема>` (ctx-режим) вместо чтения всего wiki.

**Экономия контекста:**
- Без Hermes: 8 файлов × 2k = 16k токенов
- С Hermes: 3 файла × 2k = 6k токенов
- **Экономия: 62.5%**

**Как работает:**
1. Извлечь тему из задачи (vexflow, audio, pixi, ...)
2. Вызвать `/wiki <тема>` → получить только релевантные файлы (ctx-режим)
3. Передать контент в Plan-агента
4. Plan-агент создаёт план с учётом known-issues.md (если был возвращен)

**Ключевые слова для извлечения темы:**
```
vexflow, score, print, fullscreen → vexflow
audio, synth, stream, player, sync_map → audio
pixi, layer, render, webgl → pixi
supabase, database, schema → supabase
layout, geometry, coordinates, wheel → layout
connections, links, influence, teacher → connections
eras, colors, lod → eras
medusa, acoustic_lens → medusa
bugs, issues, problems → bugs
minimax, wav, music21 → minimax
typescript, react, eslint → typescript
```
